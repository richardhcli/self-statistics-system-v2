Here is the comprehensive summary, changelog, and documentation updates for the recent plugin and backend architectural shifts.

### Changelog

**Authentication & Security**

* Transitioned the authentication architecture from a custom database API key model to utilizing Firebase Custom Tokens.
* Enforced Google Identity Bearer token validation via `middleware.ts`.
* Migrated from local `.env` variables to Google Cloud Secret Manager using `defineSecret()` for keys like `GOOGLE_AI_API_KEY`.
* Updated Google Cloud IAM permissions by granting the Default Compute service account the `Service Account Token Creator` role to mint custom tokens.

**Backend & SDK Architecture**

* Abstracted the Google Identity lifecycle into a universal, platform-agnostic `@self-stats/plugin-sdk` that handles automatic ID token refreshes.
* Unified duplicate webhook endpoints into a single REST endpoint (`apiRouter`) to handle all external integration traffic.
* Configured CORS within the Gen 2 Firebase `onRequest` function to specifically accept preflight requests from the `app://obsidian.md` protocol.
* Resolved Firestore `undefined` value crashes by enabling `ignoreUndefinedProperties: true` in the database initialization.

**Build & Deployment**

* Implemented an "Isolated Dist" build step using `build.mjs` to strip `workspace:*` dependencies from the `package.json` prior to deployment, preventing Google Cloud Build crashes.
* Created a `sync-secrets.mjs` pipeline to automate pushing local variables to Google Cloud Secret Manager during deployment.

**Obsidian Plugin UX/UI**

* Added native Obsidian plugin settings UI to accept the Setup Code and securely save the Refresh Token to Obsidian's `data.json` via the SDK's `StorageAdapter`.
* Refactored the journal injection format to utilize rich, collapsible markdown callouts displaying precise node-by-node EXP gains.
* Integrated a DataviewJS dashboard for in-note execution.

### Compartmentalized Developer Documentation

#### 1. Authentication Pipeline (Firebase Custom Tokens)

The system requires external plugins to securely communicate with the Firebase backend without utilizing web browser OAuth popups.

* **Phase 1 (Minting):** The authenticated React web app requests a 1-hour Custom Token (the "Setup Code") from the backend via `admin.auth().createCustomToken(uid)`.
* **Phase 2 (Exchange):** The plugin developer passes this code to the `@self-stats/plugin-sdk`. The SDK intercepts the string and executes an HTTP POST to the Google Identity Toolkit REST API. Google responds with a short-lived ID Token and a permanent Refresh Token.
* **Phase 3 (Auto-Refresh):** The SDK stores the Refresh Token locally. When executing backend requests, the SDK's internal fetch interceptor automatically negotiates a fresh ID Token if the current one has expired.

#### 2. Monorepo Firebase Deployment (Isolated Dist)

Deploying Firebase Functions from a monorepo that uses `workspace:*` local dependencies causes standard Cloud Build processes to fail.

* **Pre-bundling:** The codebase utilizes `esbuild` to compile the backend and all imported shared packages (e.g., `@self-stats/progression-system`) into a single `dist/index.js` file.
* **Isolation Script:** A `build.mjs` script programmatically generates a clean `package.json` inside the `dist/` folder that omits local workspace linkages. The Firebase CLI is configured to deploy directly from this `dist/` directory, successfully bypassing strict package resolution errors.

#### 3. Secret Management

Standard `.env` files are insufficient for Gen 2 Firebase Functions running on Cloud Run.

* **Binding:** Secrets are bound to endpoints using `defineSecret("GOOGLE_AI_API_KEY")`.
* **Synchronization:** Prior to deployment, a `sync-secrets.mjs` script reads the local `.env` and pushes the credentials directly to Google Cloud Secret Manager via the Firebase CLI.
